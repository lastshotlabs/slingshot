description: Overrides and manual wiring that live beneath generated package and entity behavior

The docs are intentionally optimized for the normal app-side path:

- app roots
- packages
- entities
- built-in events and realtime

When generated behavior needs tweaking, the framework exposes seams beneath it so you can
override one piece without abandoning the rest.

For framework-level plugin authoring (auth, persistence, queues, custom middleware), see the
[Plugin Interface](/slingshot/authoring/plugin-interface/) — that's a parallel first-class
authoring tier, not a seam below the package model.

## Seams within the package model

1. Customize route or entity policy before rewriting the feature
2. Override generated entity behavior before adding new routes around it
3. Add package routes beside an entity for endpoints that don't map to CRUD
4. Use manual adapter wiring only when the provided wiring modes do not fit

## Framework-level plugins

If what you're building needs lifecycle hooks, async bootstrap, or registrar registrations,
you are writing a `SlingshotPlugin`, not extending a package. See
[Plugin Interface](/slingshot/authoring/plugin-interface/).

## Use these deliberately

The point of these seams is not to bypass the framework at the first inconvenience.

They exist so you can:

- keep the normal path for most of the app
- isolate custom behavior where it is actually needed
- avoid inventing a fully custom architecture when one targeted override is enough
