The docs are intentionally optimized for the normal path:

- app roots
- packages
- entities
- built-in events and realtime

But real apps eventually hit cases that do not fit the default path cleanly.

## Reach for these in order

1. Customize route or entity policy before rewriting the feature
2. Override generated entity behavior before abandoning the entity model
3. Add package routes beside an entity before dropping to raw plugins
4. Use raw plugins only when package-first composition is genuinely the wrong layer
5. Use manual adapter wiring only when the provided wiring modes do not fit

## Use them deliberately

The point of these seams is not to bypass the framework at the first inconvenience.

They exist so you can:

- keep the normal path for most of the app
- isolate custom behavior where it is actually needed
- avoid inventing a fully custom architecture when one targeted override is enough
