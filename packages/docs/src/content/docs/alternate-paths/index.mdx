Slingshot still supports flows that should not lead the public docs narrative.

This section is where they belong.

## What is primary vs alternate

Primary:

- code-first app roots
- package-first composition
- code-first entity authoring

Alternate:

- manifest authoring
- older or lower-level plugin-only flows
- compatibility-oriented docs that remain supported but should not define the mental model

## Why this split exists

The framework has strong code-first surfaces now. The docs should teach those surfaces directly
instead of making them feel like the advanced path.

That does not mean alternate paths are deprecated. It means they should be documented honestly:

- supported
- useful in some situations
- not the main way new users should learn the framework
