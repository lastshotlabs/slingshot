The main docs teach the code-first path: `createServer(...)`, packages, entities, typed routes.

That is not the only way to use Slingshot.

## Manifest authoring

The manifest is a JSON file that assembles your app declaratively. Use it when:

- you want zero-code app assembly for standard plugin combinations
- your app is mostly configuration, not custom authoring
- you want a serialized app root that maps cleanly to deployment tooling

```json title="app.manifest.json"
{
  "manifestVersion": 1,
  "port": 3000,
  "security": {
    "signing": { "secret": "${secret:JWT_SECRET}" }
  },
  "plugins": [{ "plugin": "slingshot-auth" }]
}
```

Custom behavior (middleware, event handlers, tenant resolvers) goes in `slingshot.handlers.ts`.
The manifest references these functions by name, and the framework auto-loads them at boot.

Both paths produce identical runtime behavior. The manifest is not deprecated — it is a
real authoring surface for apps that fit the declarative model.

See [Manifest Authoring](/slingshot/alternate-paths/manifest-authoring/) for the full walkthrough
and [Manifest vs Code](/slingshot/manifest-vs-code/) for a side-by-side comparison.
