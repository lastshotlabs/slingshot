A manifest app uses `app.manifest.json` as its root instead of a TypeScript file calling
`createServer(...)`. The framework reads the manifest, wires plugins, and boots the server.

## A minimal manifest

```json title="app.manifest.json"
{
  "manifestVersion": 1,
  "port": 3000,
  "security": {
    "signing": { "secret": "${secret:JWT_SECRET}" }
  },
  "plugins": [{ "plugin": "slingshot-auth" }]
}
```

```bash
slingshot start
```

That gives you a running server with auth routes, session management, and OpenAPI docs.

## Adding custom behavior

Custom middleware, event handlers, tenant resolvers, and lifecycle hooks live in
`slingshot.handlers.ts`. The manifest references them by name:

```json title="app.manifest.json"
{
  "manifestVersion": 1,
  "port": 3000,
  "tenantResolver": "resolveTenant",
  "plugins": [{ "plugin": "slingshot-auth" }]
}
```

```typescript title="slingshot.handlers.ts"
export function resolveTenant(c: Context): string | undefined {
  return c.req.header('x-tenant-id') ?? undefined;
}
```

The framework auto-loads the handlers file and registers its exports before boot.

## When to use the manifest

Use it when your app is mostly **assembly and configuration**: selecting plugins, wiring
databases, setting security options. The manifest handles that cleanly without boilerplate.

Switch to the code-first app root when your app needs:

- package-first authoring with `definePackage(...)`
- typed routes with `domain(...)` and `route.*(...)`
- entity definitions with `defineEntity(...)`
- code-level escape hatches or dynamic composition

## Both paths produce the same runtime

A manifest app and a code-first app run the same plugin lifecycle, the same middleware
pipeline, and the same route registration. The difference is how you express the assembly step.

<Aside>
  All built-in plugins are fully configurable via the manifest. See the [Manifest
  Schema](/slingshot/internals/manifest/) for the complete config surface.
</Aside>
