---
title: Context and Request Model
description: App-scoped runtime state, request-scoped identity, and where to read each one
---

Slingshot separates app-scoped state from request-scoped state.

That distinction matters because the framework supports multiple app instances in one process, and
because auth, events, persistence, and realtime all cross route boundaries.

## App-scoped state: `SlingshotContext`

Use `getContext(app)` when you need runtime state for a specific assembled app instance.

```typescript
import { createApp, getContext } from '@lastshotlabs/slingshot';

const { app } = await createApp({ packages: [] });
const ctx = getContext(app);

ctx.events.publish('app:ready', { plugins: [] });
```

`SlingshotContext` is where the framework keeps:

- the frozen resolved config
- the event bus and event publisher
- persistence handles
- plugin and package runtime state
- runtime WebSocket state
- cleanup helpers like `clear()` and `destroy()`

## Request-scoped state

Use the route context for request data and `getActor(...)` when you need the authenticated actor.

```typescript
import { createRouter, getActor } from '@lastshotlabs/slingshot';

export const router = createRouter();

router.get('/me', c => {
  const actor = getActor(c);
  return c.json({ actorId: actor.id });
});
```

Request-scoped data is where things like params, query, headers, request id, and the authenticated
actor belong.

## What belongs where

| Use case                             | Read from                              |
| ------------------------------------ | -------------------------------------- |
| Authenticated caller for one request | `getActor(c)` or route handler context |
| Event publishing infrastructure      | `SlingshotContext`                     |
| Shared persistence handles           | `SlingshotContext`                     |
| WebSocket room state                 | `SlingshotContext.ws`                  |
| Route params and JSON body           | request or typed route context         |
| App config                           | `SlingshotContext.config`              |

## Common mistakes

- Do not stash context in a module-global variable. Use `getContext(app)` for the specific app
  instance you are working with.
- Do not treat request identity as app-scoped state.
- Do not mutate `ctx.config`. It is frozen on purpose.

## Tests

`ctx.clear()` resets in-memory state between tests. `ctx.destroy()` tears down connections and
runtime resources when the app is done.
