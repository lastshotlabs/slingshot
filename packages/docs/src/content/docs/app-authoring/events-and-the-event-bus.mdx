title: Events and the Event Bus
description: Typed events, canonical publishing, scopes, exposure, and app-level transport

The event bus is part of the core application model.

Every Slingshot app gets one. If you do not supply `eventBus`, `createApp()` creates a default
in-process adapter for you.

These examples extend the [starter app](/slingshot/app-authoring/starter-app/) with a publication
event for blog posts.

## The model

- `defineEvent(...)` defines the event contract
- `ctx.events.publish(...)` is the canonical publish path
- `bus.on(...)` is the subscription surface
- event definitions control exposure and scope for SSE, webhooks, and external consumers

## Define an event once

```typescript title="src/blog/events.ts"
import type { SlingshotPlugin } from '@lastshotlabs/slingshot-core';
import { defineEvent } from '@lastshotlabs/slingshot-core';

declare module '@lastshotlabs/slingshot-core' {
  interface SlingshotEventMap {
    'blog:post.published': {
      postId: string;
      authorId: string;
      title: string;
      slug: string;
    };
  }
}

export const blogEventsPlugin: SlingshotPlugin = {
  name: 'blog-events',
  setupMiddleware({ events }) {
    events.register(
      defineEvent('blog:post.published', {
        ownerPlugin: 'blog',
        exposure: ['client-safe', 'user-webhook'],
        resolveScope(payload, _ctx) {
          return {
            userId: payload.authorId,
            actorId: payload.authorId,
            resourceType: 'post',
            resourceId: payload.postId,
          };
        },
      }),
    );
  },
};
```

The definition is not ceremony. It controls validation, exposure, and scope resolution for every
downstream transport that depends on the event model.

## Publish through `ctx.events`

```typescript title="src/publish-post.ts"
// @skip-typecheck
import { createApp, getContext } from '@lastshotlabs/slingshot';
import { blogEventsPlugin } from './blog/events';
import { blogPackage } from './blog/package';

const { app } = await createApp({
  packages: [blogPackage],
  plugins: [blogEventsPlugin],
});
const ctx = getContext(app);

ctx.events.publish('blog:post.published', {
  postId: 'post-123',
  authorId: 'user-123',
  title: 'Shipping the docs rewrite',
  slug: 'shipping-the-docs-rewrite',
});
```

This is the canonical path because it produces the framework event envelope, applies the
registered event definition, and keeps SSE and webhook filtering coherent.

Use raw `bus.emit(...)` only for lower-level or transport-specific behavior.

## Subscribe in `setupPost`

```typescript title="src/blog/subscribers.ts"
import type { SlingshotPlugin } from '@lastshotlabs/slingshot';

export const analyticsPlugin: SlingshotPlugin = {
  name: 'analytics',
  setupPost({ bus }) {
    bus.on('blog:post.published', async payload => {
      await savePublicationAudit(payload.postId, payload.slug);
    });
  },
};

declare function savePublicationAudit(postId: string, slug: string): Promise<void>;
```

`setupPost` is the safe point for subscriptions because route and runtime registration is complete.

## Exposure and scope are first-class

Event definitions can mark events as:

- `internal`
- `client-safe`
- `tenant-webhook`
- `user-webhook`
- `app-webhook`
- `connector`

That is how one event model fans out into multiple delivery surfaces without requiring each surface
to invent its own allowlist rules.

## Security-sensitive events

Some namespaces are intentionally not browser-deliverable. `SECURITY_EVENT_TYPES` and related
framework rules exist so transports like SSE can filter sensitive events safely.

## Scaling the bus

The default in-process adapter is correct for a single process. At scale, supply a shared bus:

```typescript
import { createServer } from '@lastshotlabs/slingshot';

await createServer({
  eventBus: mySharedEventBus,
});

declare const mySharedEventBus: import('@lastshotlabs/slingshot-core').SlingshotEventBus;
```

That decision affects cross-instance event delivery, webhook fan-out, and any package behavior that
depends on events crossing instance boundaries.

## The default, the customization, and the escape hatch

- default: let `createApp()` create the in-process bus and publish through `ctx.events`
- customize: register explicit event definitions with `defineEvent(...)` and subscribe in
  `setupPost`
- escape hatch: provide your own `eventBus` implementation when the app needs shared multi-instance
  delivery

## This is why the bus is a core page

The event bus is not a plugin-authoring side topic. It is the shared event transport behind:

- package-to-package side effects
- SSE
- webhook delivery
- search sync
- security and lifecycle events
