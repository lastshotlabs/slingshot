Slingshot should be learned from the app root inward.

Start with `createServer()` or `createApp()`. Then learn the top-level config, the runtime context,
the event bus, realtime transports, and the package-first composition model those apps are built
from.

## The canonical shape

```typescript title="src/index.ts"
import { createServer, definePackage, domain, route } from '@lastshotlabs/slingshot';

const healthPackage = definePackage({
  name: 'health',
  domains: [
    domain({
      name: 'health',
      basePath: '/health',
      routes: [
        route.get({
          path: '/',
          auth: 'none',
          summary: 'Health check',
          handler: ({ respond }) => respond.json({ ok: true }),
        }),
      ],
    }),
  ],
});

await createServer({
  port: 3000,
  packages: [healthPackage],
});
```

That one file already demonstrates the core model:

- the app root is code
- packages are the canonical composition surface
- routes, auth, permissions, events, OpenAPI, and runtime concerns assemble around that root

## Full core feature map

Use this when you want to understand the full feature set of the framework core rather than only the
happy path.

- app assembly: `createApp()` and `createServer()`
- top-level app config: routes, schemas, security, db, jobs, tenancy, logging, metrics,
  observability, validation, upload, versioning, packages, plugins, event bus, WebSockets, secrets,
  runtime, permissions
- runtime context: config, events, persistence, plugin/package state, WebSocket state
- request pipeline: middleware, auth, permissions, rate limits, idempotency, event publication
- realtime: WebSockets and SSE
- package-first composition: packages, domains, typed routes, capabilities, entity refs
- entity system: generated CRUD, named operations, storage backends, route overrides, extra routes
