---
title: Packages and Plugins
description: Why packages are the canonical composition surface and when to use raw plugins
---

Slingshot has two composition mechanisms: **packages** (the normal path) and **plugins**
(the escape hatch). Most app code should be packages.

## Packages: the normal path

A package is a self-contained feature module. It owns entities, routes, middleware, and
cross-package contracts:

```typescript
// @skip-typecheck
import { createServer } from '@lastshotlabs/slingshot';
import { blogPackage } from './packages/blog';
import { commentPackage } from './packages/comments';

await createServer({
  port: 3000,
  packages: [blogPackage, commentPackage],
});
```

Packages give you:

- **Entity modules** â€” generated CRUD + operations from `defineEntity`
- **Domain routes** â€” typed custom routes from `domain()` + `route.*()`
- **Named middleware** â€” scoped to the package
- **Capabilities** â€” typed cross-package service contracts
- **Entity refs** â€” cross-package data access without coupling
- **Dependency ordering** â€” packages boot in the right order

The app root reads like a table of contents. Feature logic lives in packages.

## Plugins: the escape hatch

Plugins give you direct lifecycle hooks. Use them for infrastructure concerns that do not
fit the package model:

```typescript
// @skip-typecheck
import type { SlingshotPlugin } from '@lastshotlabs/slingshot-core';

const metricsPlugin: SlingshotPlugin = {
  name: 'metrics',
  setupMiddleware({ app }) {
    // Runs before any routes â€” global timing middleware
    app.use('*', async (c, next) => {
      const start = Date.now();
      await next();
      console.log(`${c.req.method} ${c.req.path} ${Date.now() - start}ms`);
    });
  },
  setupPost({ ctx }) {
    // Runs after all routes are registered
    console.log('Server ready');
  },
  teardown() {
    // Runs on graceful shutdown
    console.log('Shutting down');
  },
};

await createServer({
  port: 3000,
  plugins: [metricsPlugin],
  packages: [blogPackage],
});
```

## When to use which

| You are building...                        | Use                       |
| ------------------------------------------ | ------------------------- |
| A feature with entities and routes         | Package                   |
| A feature with only custom routes          | Package                   |
| A cross-cutting middleware (logging, CORS) | Plugin                    |
| A framework integration (metrics, tracing) | Plugin                    |
| Auth (sessions, OAuth, MFA)                | `createAuthPlugin`        |
| Permissions (RBAC, grants)                 | `createPermissionsPlugin` |

The built-in auth and permissions plugins are pre-built â€” you configure them, not write them.
Custom infrastructure work is where raw plugins make sense.

## How they relate

Packages compile to plugins under the hood. When you call `createServer({ packages: [...] })`,
the framework:

1. Validates package dependencies
2. Sorts packages in dependency order
3. Converts each package into an internal plugin
4. Runs all plugins (both raw plugins and package-derived) in order

This is an implementation detail, but it explains why packages and plugins coexist: they
share the same lifecycle but present different authoring surfaces.

## Mixing packages and plugins

You can use both in the same app:

```typescript
// @skip-typecheck
await createServer({
  port: 3000,
  plugins: [
    metricsPlugin,                // infrastructure: raw plugin
    createAuthPlugin({ ... }),    // auth: pre-built plugin
  ],
  packages: [
    blogPackage,                  // feature: package
    commentPackage,               // feature: package
    adminPackage,                 // feature: package
  ],
});
```

Plugins run first (in order), then packages boot in dependency order.
