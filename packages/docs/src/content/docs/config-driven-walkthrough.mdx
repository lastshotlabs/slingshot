description: Define an entity once. Get routes, adapters, events, and permissions without writing the wiring.

The core Slingshot authoring pattern is now:

1. define an entity
2. define operations
3. mount them in a package
4. compose the app

This walkthrough uses the primary code-first path. Manifest-backed authoring still exists, but it is not the default framing here.

## 1. The entity carries the core model

```typescript
import { defineEntity, field, index } from '@lastshotlabs/slingshot';

export const Thread = defineEntity('Thread', {
  namespace: 'community',
  fields: {
    id: field.string({ primary: true, default: 'uuid' }),
    containerId: field.string(),
    authorId: field.string(),
    title: field.string(),
    body: field.string({ optional: true }),
    status: field.enum(['draft', 'published', 'deleted'] as const, { default: 'draft' }),
    locked: field.boolean({ default: false }),
    pinned: field.boolean({ default: false }),
    createdAt: field.date({ default: 'now' }),
    updatedAt: field.date({ default: 'now', onUpdate: 'now' }),
  },
  indexes: [
    index(['containerId', 'createdAt'], { direction: 'desc' }),
    index(['authorId', 'createdAt'], { direction: 'desc' }),
  ],
  routes: {
    defaults: { auth: 'userAuth' },
    get: { auth: 'none' },
    list: { auth: 'none' },
    create: {
      permission: {
        requires: 'community:container.write',
        scope: { resourceType: 'community:container', resourceId: 'param:containerId' },
      },
      event: {
        key: 'community:thread.created',
        payload: ['id', 'containerId', 'authorId', 'title'],
        scope: { resourceType: 'community:container', resourceId: 'record:containerId' },
        exposure: ['client-safe'],
      },
      middleware: ['banCheck', 'autoMod'],
    },
  },
});
```

One definition can carry:

- field structure
- indexes
- generated route defaults
- auth and permission config
- event emission metadata
- named middleware references

## 2. Operations express domain behavior

CRUD comes from the entity automatically. Operations are for behavior that should become typed adapter methods and managed HTTP routes.

```typescript
import { defineOperations, op } from '@lastshotlabs/slingshot';

export const threadOperations = defineOperations(Thread, {
  byContainer: op.lookup({
    fields: { containerId: 'param:containerId' },
    filter: { status: { $ne: 'deleted' } },
    returns: 'many',
    sort: 'createdAt',
    direction: 'desc',
    paginate: true,
  }),

  publish: op.transition({
    field: 'status',
    from: 'draft',
    to: 'published',
    match: { id: 'param:id' },
    returns: 'entity',
  }),

  lock: op.fieldUpdate({
    match: { id: 'param:id' },
    set: ['locked'],
  }),

  addTag: op.arrayPush({
    field: 'tags',
    value: 'input:tag',
  }),

  countByContainer: op.aggregate({
    groupBy: 'containerId',
    compute: { total: 'count' },
    filter: { status: 'published' },
  }),
});
```

Each operation can produce:

- a typed adapter method
- an auto-mounted route in the managed entity shell
- validation and OpenAPI metadata

## 3. The package owns assembly

`definePackage(...)` is the default top-level authoring surface.

```typescript
import type { MiddlewareHandler } from 'hono';
import { createApp, definePackage, domain, route } from '@lastshotlabs/slingshot';
import { entity } from '@lastshotlabs/slingshot';

declare function createBanCheckMiddleware(): MiddlewareHandler;
declare function createAutoModMiddleware(): MiddlewareHandler;

export const threadsPackage = definePackage({
  name: 'threads',
  mountPath: '/community',
  entities: [
    entity({
      config: Thread,
      operations: threadOperations,
    }),
  ],
  middleware: {
    banCheck: createBanCheckMiddleware(),
    autoMod: createAutoModMiddleware(),
  },
  domains: [
    domain({
      name: 'analytics',
      basePath: '/analytics',
      routes: [
        route.get({
          path: '/summary',
          async handler(ctx) {
            return ctx.respond.json({ packageName: ctx.packageName });
          },
        }),
      ],
    }),
  ],
});

await createApp({
  packages: [threadsPackage],
});
```

That package now owns:

- entity-backed CRUD and operation routes
- package-owned custom routes under `domain(...).routes`
- named middleware usable by those routes
- mount path and dependency composition

## 4. Use the entity shell before a manual router

If a route still belongs to one entity, prefer the entity shell first.

Use:

- generated CRUD routes
- generated operation routes
- `extraRoutes`
- generated route `overrides`

Only drop lower when the route no longer belongs to one entity or no longer fits the managed route shell.

## 5. Actor-first request context

The public identity model is actor-first.

Use bindings like:

- `ctx:actor.id`
- `ctx:actor.tenantId`
- `ctx:actor.claims.orgId`

Example:

```jsonc
{
  "routes": {
    "defaults": { "auth": "userAuth" },
    "dataScope": { "field": "authorId", "from": "ctx:actor.id" },
  },
}
```

Legacy aliases such as `ctx:authUserId` and `ctx:tenantId` still resolve for compatibility, but they are not the default authoring model.

## 6. What gets generated

From one entity plus operations, the managed layer can mount:

- `GET /threads`
- `POST /threads`
- `GET /threads/:id`
- `PATCH /threads/:id`
- `DELETE /threads/:id`
- named operation routes like `/threads/by-container` or `/threads/:id/publish`

The exact shape depends on your operations and route config, but the point is that the framework owns the repetitive wiring.

## 7. Reach for tools in this order

1. Existing feature packages
2. Your own package-first entity/domain authoring
3. Package domain routes for orchestration and custom endpoints
4. Escape hatches like `routesDir`, raw Hono, manual adapters, raw SQL/Drizzle, or raw plugins
