The primary Slingshot authoring path is now package-first and code-first.

You define:

- entities with `defineEntity(...)`
- domain behavior with `defineOperations(...)`
- package composition with `definePackage(...)`
- package-owned custom endpoints with `domain(...)` and `route.*(...)`

From those declarations, Slingshot derives adapters, routes, validation, OpenAPI, and runtime wiring without forcing you down to raw plugins or ad hoc router glue.

## The default stack

For most teams, the intended path is:

1. Define entities with `defineEntity(...)`
2. Define domain behavior with `defineOperations(...)`
3. Mount them in packages with `entity(...)`
4. Add package-owned custom endpoints with `domain({ routes: [...] })`
5. Compose the app with `createApp({ packages })`

That gives you a managed route shell, actor-first request identity, generated OpenAPI metadata, and a package boundary that stays coherent as the app grows.

## What gets generated

From those declarations, Slingshot can generate or assemble:

- storage adapters
- CRUD and operation-backed HTTP routes
- route validation and OpenAPI metadata
- policy, auth, middleware, and event wiring
- typed package capability contracts

The goal is not “everything is magic.” The goal is that the repetitive wiring is framework-owned and the domain decisions stay in one place.

## Where escape hatches start

The config-driven stack is the default, not the only option.

When you need lower-level control, you can drop to:

- `routesDir` + `createRoute(...)` + `router.openapi(...)`
- custom factories or manual adapter wiring
- raw SQL or Drizzle-backed repo code
- package domain routes for orchestration or reporting
- raw `SlingshotPlugin` lifecycle hooks

Those escape hatches are documented explicitly in [Escape Hatches](/slingshot/config-driven/escape-hatches/).

## Manifest support

Manifest and JSON-backed authoring still exist, but they are no longer the primary mental model for this section.

Use the manifest path when:

- you are working with manifest-driven server bootstrap
- you need a machine-editable declarative artifact
- you are integrating with tooling that emits manifest config directly

If you need that path, start here:

- [Manifest vs Code](/slingshot/manifest-vs-code/)
