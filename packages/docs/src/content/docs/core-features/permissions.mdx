Auth tells you _who_ is making the request. Permissions tell you _whether they are allowed_
to do what they are asking.

In Slingshot, these are separate concerns. Auth gives you an actor. Permissions decide what
that actor can access.

## The simplest version: role checks

If all you need is "admins can delete, everyone else can read," use route-level auth:

```typescript
const Post = defineEntity('Post', {
  namespace: 'blog',
  fields: {
    id: field.string({ primary: true, default: 'uuid' }),
    title: field.string(),
    body: field.string(),
    authorId: field.string(),
  },
  routes: {
    defaults: { auth: 'userAuth' },
    delete: { auth: 'userAuth', middleware: ['requireAdmin'] },
  },
});
```

This works when permissions are purely role-based. Every authenticated user can read, only
admins can delete.

## Resource-level permissions

When you need "users can only edit _their own_ posts" or "org members can access _their
org's_ data," add the permissions plugin:

```bash
bun add @lastshotlabs/slingshot-permissions
```

```typescript
import { createServer } from '@lastshotlabs/slingshot';
import { createAuthPlugin } from '@lastshotlabs/slingshot-auth';
import {
  createPermissionsPlugin,
  createAuthGroupResolver,
} from '@lastshotlabs/slingshot-permissions';
import { getAuthRuntimePeerOrNull } from '@lastshotlabs/slingshot-core';

await createServer({
  port: 3000,
  security: { signing: { secret: process.env.JWT_SECRET! } },
  plugins: [
    createAuthPlugin({ auth: { roles: ['user'] } }),
    createPermissionsPlugin({
      groupResolver: (pluginState) =>
        createAuthGroupResolver(() => getAuthRuntimePeerOrNull(pluginState)),
    }),
  ],
  packages: [
    /* your packages */
  ],
});
```

## Permissions in entity route policy

The most common use case — declare what is required per route, the framework enforces it:

```typescript
const Post = defineEntity('Post', {
  namespace: 'blog',
  fields: {
    id: field.string({ primary: true, default: 'uuid' }),
    title: field.string(),
    body: field.string(),
    authorId: field.string(),
  },
  routes: {
    defaults: { auth: 'userAuth' },
    list: { auth: 'none' },
    get: { auth: 'none' },
    create: {
      permission: { requires: 'post:create', scope: { resourceType: 'post' } },
    },
    update: {
      permission: { requires: 'post:update', ownerField: 'authorId', scope: { resourceType: 'post' } },
    },
    delete: {
      permission: { requires: 'post:delete', ownerField: 'authorId', or: 'post:moderate', scope: { resourceType: 'post' } },
    },
  },
});
```

`ownerField: 'authorId'` checks that `record.authorId === actor.id`. No extra code needed —
the framework fetches the record and checks ownership automatically.

## Permissions in custom routes

For package routes, the same `permission` option applies:

```typescript
route.delete({
  path: '/posts/:id/archive',
  auth: 'userAuth',
  permission: { requires: 'post:archive', scope: { resourceType: 'post' } },
  handler: async ({ params, respond }) => {
    // Permission check already passed — safe to proceed
    await archivePost(params.id);
    return respond.noContent();
  },
});
```

For checks that depend on runtime values, inject the evaluator via `services` and use
`actor.id` from the handler context directly:

```typescript
import { getPermissionsState } from '@lastshotlabs/slingshot-core';

const { evaluator } = getPermissionsState(app);

domain({
  name: 'posts',
  basePath: '/posts',
  services: { evaluator },
  routes: [
    route.delete({
      path: '/:id',
      auth: 'userAuth',
      handler: async ({ actor, params, services, respond }) => {
        const post = await getPost(params.id);

        const allowed = await services.evaluator.can(
          { subjectId: actor.id!, subjectType: 'user' },
          'post:delete',
          { tenantId: actor.tenantId ?? undefined, resourceType: 'post', resourceId: post.id },
        );

        if (!allowed) return respond.json({ error: 'Forbidden' }, 403);
        await deletePost(post.id);
        return respond.noContent();
      },
    }),
  ],
});
```

## Permissions in WebSocket guards

The same evaluator works for realtime access control:

```typescript
onRoomSubscribe: async (ws, room) => {
  const allowed = await evaluator.can(
    { subjectId: ws.data.actorId, subjectType: 'user' },
    'room:join',
    { tenantId: ws.data.tenantId ?? undefined, resourceType: 'room', resourceId: room },
  );
  return allowed;
};
```

One permission model across HTTP routes, entities, and realtime.
