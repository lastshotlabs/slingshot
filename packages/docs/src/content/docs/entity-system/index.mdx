title: Entity System
description: The canonical authoring path for data-shaped domains in Slingshot

The entity system is how you author data-shaped domains the same way the framework's own packages
do.

It is code-first, package-friendly, and designed around generated behavior with explicit
customization seams.

## The canonical flow

```typescript
import { defineOperations, definePackage, defineEntity, entity, field, op } from '@lastshotlabs/slingshot';

const Post = defineEntity('Post', {
  namespace: 'blog',
  fields: {
    id: field.string({ primary: true, default: 'uuid' }),
    authorId: field.string(),
    title: field.string(),
    status: field.enum(['draft', 'published'] as const, { default: 'draft' }),
  },
  routes: {
    defaults: { auth: 'userAuth' },
    list: { auth: 'none' },
  },
});

const PostOps = defineOperations(Post, {
  publish: op.transition({
    field: 'status',
    from: 'draft',
    to: 'published',
    match: { id: 'param:id' },
  }),
});

export const blogPackage = definePackage({
  name: 'blog',
  entities: [
    entity({
      config: Post,
      operations: PostOps,
    }),
  ],
});
```

## What the entity system gives you

- one canonical entity definition
- generated CRUD routes
- named operations beyond CRUD
- route policy for auth, permissions, rate limiting, idempotency, and events
- swappable storage backends
- customization seams for overrides and extra routes
- WebSocket channel declarations and other advanced wiring when needed
