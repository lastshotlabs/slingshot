Operations are how the entity system models behavior beyond basic CRUD.

The examples on this page continue the [starter app](/slingshot/app-authoring/starter-app/) and
its `Post` entity.

## Define operations next to the entity

```typescript title="src/blog/post-operations.ts"
import { defineOperations, op } from '@lastshotlabs/slingshot';
import { Post } from './post';

export const PostOps = defineOperations(Post, {
  byAuthor: op.lookup({
    fields: { authorId: 'param:authorId' },
    returns: 'many',
  }),
  bySlug: op.lookup({
    fields: { slug: 'param:slug' },
    returns: 'one',
  }),
  existsBySlug: op.exists({
    fields: { slug: 'param:slug' },
  }),
  publish: op.transition({
    field: 'status',
    from: 'draft',
    to: 'published',
    match: { id: 'param:id' },
  }),
  retitle: op.fieldUpdate({
    match: { id: 'param:id' },
    set: ['title'],
  }),
  searchPublished: op.search({
    fields: ['title', 'body'],
    filter: { status: 'published' },
  }),
});
```

That gives the package:

- adapter methods like `byAuthor(...)`, `bySlug(...)`, and `publish(...)`
- generated named-operation routes
- generated docs for those routes
- a single place to evolve domain behavior without scattering helpers

## What the generated routes usually look like

For the `PostOps` above, the generated route shell usually includes routes like:

- `GET /blog/posts/by-author/:authorId`
- `GET /blog/posts/by-slug/:slug`
- `GET /blog/posts/exists-by-slug/:slug`
- `POST /blog/posts/:id/publish`
- `PATCH /blog/posts/:id/retitle`

## What operations are for

Use operations when the behavior is still adapter-shaped and belongs to the domain model:

- lookups
- existence checks
- transitions
- field updates
- search
- aggregates
- batches
- transactions
- derived or computed behaviors

## Why operations matter to the docs structure

They are the point where the entity system stops being "CRUD generation" and becomes a real domain
runtime.

That is why operations need their own first-class docs home instead of being buried inside generic
entity examples.

## Where they show up

Operations affect:

- generated adapter methods
- generated named-op routes
- OpenAPI
- route policy
- backend delegation such as search-provider integration

## Use the right level

If the behavior belongs to the entity adapter surface, make it an operation.

If it is really a package-owned route with different semantics, make it a domain route instead.
