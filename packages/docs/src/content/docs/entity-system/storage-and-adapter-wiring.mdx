---
title: Storage and Adapter Wiring
description: Swappable backends, factory composition, and standard vs manual wiring
---

The storage story is one of the core strengths of the entity system.

One entity definition can target multiple backends without changing the domain model.

The examples on this page continue the [starter app](/slingshot/app-authoring/starter-app/) and
its `Post` entity.

## The default story: `createEntityFactories(...)`

```typescript title="src/blog/post-factories.ts"
import { createEntityFactories } from '@lastshotlabs/slingshot';
import { Post } from './post';
import { PostOperations } from './post-operations';

export const postFactories = createEntityFactories(Post, PostOperations);
```

From one entity config, the framework can resolve adapters for:

- `memory`
- `redis`
- `sqlite`
- `mongo`
- `postgres`

That is the core "swappable storage" story.

## Wiring modes in `entity(...)`

Entity modules support three wiring modes:

### `standard`

The default. Let the framework resolve the adapter from the entity config and active store.

```typescript title="src/blog/package.ts"
entity({
  config: Post,
  operations: PostOperations,
})
```

### `factories`

Provide an explicit repo-factory set, usually from `createEntityFactories(...)` or
`createCompositeFactories(...)`.

```typescript title="src/blog/package.ts"
entity({
  config: Post,
  operations: PostOperations,
  wiring: {
    mode: 'factories',
    factories: postFactories,
  },
})
```

### `manual`

Fully manual adapter construction for advanced cases.

```typescript title="src/blog/package.ts"
entity({
  config: Post,
  wiring: {
    mode: 'manual',
    buildAdapter: (storeType, infra) => {
      void storeType;
      return createMyAdapter(infra);
    },
  },
})

declare function createMyAdapter(infra: unknown): unknown;
```

## Backend swapping at the app root

The package and entity code above does not need to change when the app changes stores.

```typescript title="src/index.ts"
await createServer({
  db: {
    sqlite: './data/app.db',
  },
  packages: [blogPackage],
});
```

Later, the same package can move to a different store strategy:

```typescript title="src/index.ts"
await createServer({
  db: {
    postgres: process.env.DATABASE_URL,
    redis: process.env.REDIS_URL,
  },
  packages: [blogPackage],
});
```

That is the main point of the storage story: the domain model stays stable while persistence
changes around it.

## Composite factories

Use `createCompositeFactories(...)` when a package owns several related entities and you want one
combined factory surface for cross-entity behavior.

## Swappable storage is not just backend swapping

The advanced part of the story is that consumers can also shape:

- field naming conventions
- system-field conventions
- TTL field mapping
- default generation and update conventions

That is why the entity docs need to explain backend swapping and consumer shape hardening together.
