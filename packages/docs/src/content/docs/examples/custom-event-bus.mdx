---
title: Custom Event Bus
description: Replace the in-process event bus with BullMQ or a custom implementation for multi-instance deployments
---

The default event bus runs in-process - fast and zero-configuration for single-instance apps. Multi-instance deployments require events to cross process boundaries.

## When you need this

Use a custom event bus when:

- You are running multiple instances of your Slingshot app behind a load balancer
- You need SSE clients on one instance to receive events emitted on another
- You want durable event delivery with a message queue
- You are building a distributed system that shares events across services

For single-instance apps, the in-process bus is the right choice.

## BullMQ event bus

For durable, at-least-once delivery with retries and dead-letter queues:

```bash
bun add @lastshotlabs/slingshot-bullmq bullmq ioredis
```

### Via manifest

```json title="app.manifest.json"
{
  "manifestVersion": 1,
  "eventBus": "bullmq",
  "secrets": { "provider": "env" }
}
```

Built-in BullMQ manifest support is available through `createServerFromManifest()`.
It resolves Redis credentials from the configured secrets provider and auto-loads
`@lastshotlabs/slingshot-bullmq` only when the manifest selects `bullmq`.

Use the object form to customize retry counts or queue prefixes without writing
your own registry wiring:

```json title="app.manifest.json"
{
  "manifestVersion": 1,
  "eventBus": {
    "type": "bullmq",
    "config": {
      "prefix": "myapp:events",
      "attempts": 5
    }
  },
  "secrets": { "provider": "env" }
}
```

BullMQ-backed events survive process restarts and can be retried on failure. Use
this for anything where losing an event is unacceptable, such as payment
webhooks, notification delivery, or audit logging.

### Via registry

If you need full control over the BullMQ adapter, register your own factory:

```typescript title="src/index.ts"
import { createManifestHandlerRegistry, createServerFromManifest } from '@lastshotlabs/slingshot';
import { createBullMQAdapter } from '@lastshotlabs/slingshot-bullmq';

const registry = createManifestHandlerRegistry();
registry.registerEventBus('bullmq', () =>
  createBullMQAdapter({
    connection: { host: process.env.REDIS_HOST ?? '127.0.0.1', port: 6379 },
    prefix: 'myapp:events',
    attempts: 5,
  }),
);

await createServerFromManifest('./app.manifest.json', registry);
```

Registry event-bus names are unique. Use `'bullmq'` here only when you want to
own that built-in slot explicitly; duplicate registrations fail instead of
quietly overriding an earlier factory.

## Kafka event bus and connectors

Use `@lastshotlabs/slingshot-kafka` when Slingshot events need to cross process
boundaries through Kafka, or when you need to bridge the internal event bus to
external Kafka topics.

```bash
bun add @lastshotlabs/slingshot-kafka kafkajs
```

### Via manifest

```json title="app.manifest.json"
{
  "manifestVersion": 1,
  "handlers": "slingshot.handlers.ts",
  "eventBus": {
    "type": "kafka",
    "config": {
      "brokers": ["127.0.0.1:9092"],
      "clientId": "my-app"
    }
  },
  "kafkaConnectors": {
    "brokers": ["127.0.0.1:9092"],
    "inbound": [
      {
        "topic": "orders.created",
        "groupId": "billing-sync",
        "handler": "captureKafkaInbound"
      }
    ],
    "outbound": [
      {
        "event": "billing:invoice.created",
        "topic": "billing.invoice.created"
      }
    ]
  },
  "secrets": { "provider": "env" }
}
```

```typescript title="slingshot.handlers.ts"
export async function captureKafkaInbound(payload: unknown) {
  console.log(payload);
}
```

The built-in manifest path resolves `@lastshotlabs/slingshot-kafka` only when
the manifest selects Kafka. Broker addresses can come from
`eventBus.config.brokers`, `kafkaConnectors.brokers`, or `KAFKA_BROKERS`.

### Via code

```typescript title="src/index.ts"
import { createApp } from '@lastshotlabs/slingshot';
import { createKafkaAdapter, createKafkaConnectors } from '@lastshotlabs/slingshot-kafka';

const eventBus = createKafkaAdapter({
  brokers: ['127.0.0.1:9092'],
  clientId: 'my-app',
});

const kafkaConnectors = createKafkaConnectors({
  brokers: ['127.0.0.1:9092'],
  inbound: [
    {
      topic: 'orders.created',
      groupId: 'billing-sync',
      handler: async (payload, metadata) => {
        console.log(metadata.topic, payload);
      },
    },
  ],
  outbound: [
    {
      event: 'billing:invoice.created',
      topic: 'billing.invoice.created',
    },
  ],
});

await createApp({
  eventBus,
  kafkaConnectors,
});
```

Kafka is the right fit when another system already standardizes on Kafka, when
you need durable replayable topics, or when platform teams own the broker and
want applications to integrate at the transport boundary instead of inventing a
custom bus.

## Redis event bus

```bash
bun add ioredis
```

### Via manifest

```json title="app.manifest.json"
{
  "manifestVersion": 1,
  "eventBus": {
    "type": "redis",
    "config": {
      "channel": "slingshot:events"
    }
  },
  "secrets": { "provider": "env" }
}
```

Redis pub/sub is not a built-in manifest literal. Register a custom Redis bus
implementation under a name like `"redis"` and select it with the object form
shown above.

### Via registry

Register a custom Redis bus implementation:

```typescript title="src/index.ts"
import { createManifestHandlerRegistry, createServerFromManifest } from '@lastshotlabs/slingshot';
import { createRedisEventBus } from './eventBus.js';

const registry = createManifestHandlerRegistry();

registry.registerEventBus('redis', () => {
  return createRedisEventBus({
    url: process.env.REDIS_URL ?? 'redis://127.0.0.1:6379',
    channel: 'slingshot:events',
  });
});

await createServerFromManifest('./app.manifest.json', registry);
```

```typescript title="src/eventBus.ts"
import { Redis } from 'ioredis';
import type {
  EventEnvelope,
  SlingshotEventBus,
  SlingshotEventMap,
  SubscriptionOpts,
} from '@lastshotlabs/slingshot-core';

type RedisPayloadHandler = (payload: never) => void | Promise<void>;
type RedisEnvelopeHandler = (envelope: never) => void | Promise<void>;

class RedisEventBus implements SlingshotEventBus {
  private readonly pub: Redis;
  private readonly sub: Redis;
  private readonly handlers = new Map<string, Set<RedisPayloadHandler>>();

  constructor(
    url: string,
    private readonly channel: string,
  ) {
    this.pub = new Redis(url);
    this.sub = new Redis(url);

    this.sub.subscribe(channel);
    this.sub.on('message', (_ch, msg) => {
      const { key, payload } = JSON.parse(msg) as { key: string; payload: unknown };
      this.handlers.get(key)?.forEach(handler => handler(payload as never));
    });
  }

  emit<K extends keyof SlingshotEventMap>(event: K, payload: SlingshotEventMap[K]): void;
  emit(event: string, payload: unknown): void;
  emit(key: string, payload: unknown): void {
    this.pub.publish(this.channel, JSON.stringify({ key, payload }));
  }

  on<K extends keyof SlingshotEventMap>(
    event: K,
    listener: (payload: SlingshotEventMap[K]) => void | Promise<void>,
    opts?: SubscriptionOpts,
  ): void;
  on(event: string, listener: RedisPayloadHandler, opts?: SubscriptionOpts): void;
  on(key: string, handler: RedisPayloadHandler): void {
    if (!this.handlers.has(key)) this.handlers.set(key, new Set());
    this.handlers.get(key)?.add(handler);
  }

  onEnvelope<K extends keyof SlingshotEventMap>(
    event: K,
    listener: (envelope: EventEnvelope<K>) => void | Promise<void>,
    opts?: SubscriptionOpts,
  ): void;
  onEnvelope(event: string, listener: RedisEnvelopeHandler, opts?: SubscriptionOpts): void;
  onEnvelope(): void {}

  off<K extends keyof SlingshotEventMap>(
    event: K,
    listener: (payload: SlingshotEventMap[K]) => void,
  ): void;
  off(event: string, listener: (payload: unknown) => void): void;
  off(key: string, handler: RedisPayloadHandler): void {
    this.handlers.get(key)?.delete(handler);
  }

  offEnvelope<K extends keyof SlingshotEventMap>(
    event: K,
    listener: (envelope: EventEnvelope<K>) => void,
  ): void;
  offEnvelope(event: string, listener: (envelope: EventEnvelope) => void): void;
  offEnvelope(): void {}

  async shutdown(): Promise<void> {
    await Promise.all([this.pub.quit(), this.sub.quit()]);
  }
}

export function createRedisEventBus({
  url,
  channel,
}: {
  url: string;
  channel: string;
}): SlingshotEventBus {
  return new RedisEventBus(url, channel);
}
```

## The event bus interface

Any implementation of `SlingshotEventBus` from `slingshot-core` works:

```typescript
interface SlingshotEventBus {
  emit<K extends EventKey>(key: K, payload: EventPayload<K>): void;
  on<K extends EventKey>(key: K, handler: EventHandler<K>): void;
  off<K extends EventKey>(key: K, handler: EventHandler<K>): void;
  shutdown?(): Promise<void>;
}
```

The framework does not care what is underneath - Redis, NATS, Kafka, or a custom transport. Implement the interface and plugins, entities, and SSE endpoints continue to work. Event exposure is enforced one layer up by the registry-backed `ctx.events` publisher, not by the bus transport.
