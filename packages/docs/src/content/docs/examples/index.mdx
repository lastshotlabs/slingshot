These are the highest-signal starting points for the platform.

Use them in two different ways:

- by product shape: "which example looks like my app?"
- by learning stage: "what example should I read after the docs spine?"

All examples use in-memory adapters by default. Swapping to SQLite, Postgres, MongoDB, or Redis is
a one-line change. See [Production Databases](/slingshot/examples/production-databases/).

Source-backed example apps live in the repo under `examples/`. The current source-backed set is
`with-auth`, `config-driven-domain`, `collaboration-workspace`, `content-platform`, and
`orchestration`.

## Recommended progression

If you want the learning path to stay coherent, use examples in this order:

1. [Config-Driven Domain](/slingshot/examples/config-driven-domain/) after [Quick Start](/slingshot/quick-start/)
2. one product-shaped example that matches your app
3. a focused advanced example like [Realtime with SSE](/slingshot/examples/realtime-sse/) or
   [Custom Event Bus](/slingshot/examples/custom-event-bus/)
4. [Custom Plugin](/slingshot/examples/custom-plugin/) only when you actually need escape hatches

## Ready for production

Once you have a working app modeled after an example, the guides cover what you need to ship:

- [Security](/slingshot/guides/security/) — CORS, CSRF, rate limits, helmet, secrets rotation
- [Testing](/slingshot/guides/testing/) — unit, integration, and entity route testing patterns
- [Deployment](/slingshot/guides/deployment/) — Docker, serverless, and managed hosting
- [Multi-Tenancy](/slingshot/guides/multi-tenancy/) — tenant resolution, scoped data, and isolation
- [Horizontal Scaling](/slingshot/guides/horizontal-scaling/) — stateless workers, external event buses, and session stores

Browse all guides under [Guides](/slingshot/guides/).
