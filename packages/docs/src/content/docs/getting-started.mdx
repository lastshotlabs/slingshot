## What Slingshot is

Slingshot is a backend framework for assembling real applications out of coherent domain surfaces
instead of scattered route files and ad hoc service glue.

The core idea is simple:

- start at the app root
- compose packages
- author data-shaped domains through the entity system
- let routes, permissions, events, docs, and runtime behavior stay attached to that model

## The main mental model

Start with code:

1. `createServer()` or `createApp()`
2. top-level app config
3. package-first composition with `definePackage(...)`
4. entity authoring with `defineEntity(...)`, `defineOperations(...)`, and `entity(...)`

That is the public path the docs are optimized for.

## The core layers

### App layer

This is the framework root:

- app assembly
- runtime context
- security and middleware
- event bus
- WebSockets
- SSE
- OpenAPI and validation
- runtime and infrastructure

### Package-first authoring

This is the canonical authoring surface for app modules:

- `definePackage(...)`
- `domain(...)`
- `route.*(...)`
- `defineCapability(...)`
- `provideCapability(...)`
- `entityRef(...)`

### Entity system

This is the canonical path for data-shaped domains:

- `defineEntity(...)`
- `defineOperations(...)`
- `op.*(...)`
- `entity(...)`
- generated routes and named operations
- swappable storage and adapter wiring

### Lower-level escape hatches

These still exist:

- [raw plugins](/slingshot/authoring/plugin-interface/)
- [manual adapter wiring](/slingshot/entity-system/storage-and-adapter-wiring/#manual)
- [extra routes](/slingshot/entity-system/generated-routes-overrides-and-extra-routes/#add-an-extra-route)
- [route executor overrides](/slingshot/entity-system/generated-routes-overrides-and-extra-routes/#override-a-generated-route)

They matter, but they should not be the first thing people learn. See [Escape Hatches](/slingshot/config-driven/escape-hatches/) for the full guide.

## How the docs are organized

The docs follow the real authoring experience:

1. Core API surface and package-first composition first
2. Entity system and config-driven patterns second
3. Plugin internals and manifests only when you need them

## What not to miss

There are a few core features that must be understood early because they affect almost every real
app:

- the event bus
- WebSockets
- SSE
- packages vs plugins
- the entity storage story

Those all have first-class docs homes now.
