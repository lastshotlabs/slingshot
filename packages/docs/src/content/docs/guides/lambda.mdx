`@lastshotlabs/slingshot-runtime-lambda` wraps `SlingshotHandler` instances for AWS Lambda without
forking your application logic by transport.

## Install

```bash
bun add @lastshotlabs/slingshot-runtime-lambda
```

## Direct runtime wrapping

Use `createLambdaRuntime()` when your entry module chooses the exported Lambda handlers explicitly:

```ts
import { createLambdaRuntime } from '@lastshotlabs/slingshot-runtime-lambda';
import type { SlingshotHandler } from '@lastshotlabs/slingshot-core';

declare const processOrder: SlingshotHandler;

const lambda = createLambdaRuntime({
  manifest: './app.manifest.json',
});

export const api = lambda.wrap(processOrder, 'apigw-v2');
export const queue = lambda.wrap(processOrder, 'sqs');
export const nightly = lambda.wrap(processOrder, 'schedule');
```

Cold start runs the same manifest pipeline as `createServerFromManifest()`, then calls `createApp()`
without binding a port. Warm invocations reuse that cached context.

## Trigger matrix

Supported trigger kinds:

- `apigw`
- `apigw-v2`
- `function-url`
- `alb`
- `sqs`
- `msk`
- `kinesis`
- `dynamodb-streams`
- `s3`
- `sns`
- `eventbridge`
- `schedule`

HTTP-style triggers decode path params, query params, and request bodies into a single handler input.
Batch-style triggers extract one record per event item and run the handler once per record.

## Manifest-driven exports

Use the manifest `lambdas` section when the bindings should stay declarative:

```json title="app.manifest.json"
{
  "manifestVersion": 1,
  "handlers": "slingshot.handlers.ts",
  "lambdas": {
    "processOrderApi": {
      "handler": "processOrder",
      "trigger": "apigw-v2"
    },
    "processOrderQueue": {
      "handler": "processOrder",
      "trigger": "sqs",
      "idempotency": {
        "ttl": 3600
      }
    }
  }
}
```

Then build the export map directly:

```ts
import { createFunctionsFromManifest } from '@lastshotlabs/slingshot-runtime-lambda';

const handlers = await createFunctionsFromManifest('./app.manifest.json');

export const processOrderApi = handlers.processOrderApi;
export const processOrderQueue = handlers.processOrderQueue;
```

The important split is:

- `createServerFromManifest()` validates `lambdas` but ignores it
- `createFunctionsFromManifest()` consumes `lambdas`

That keeps the manifest as the shared source of truth for HTTP server and Lambda deployments without
mixing the runtime paths.

## Idempotency

Natural idempotency is auto-enabled for batch sources with stable record identifiers:

- SQS: `messageId`
- MSK: `topic:partition:offset`
- Kinesis: `sequenceNumber`
- DynamoDB Streams: `eventID`

HTTP triggers can also use the `Idempotency-Key` header. Override scope, TTL, or fingerprint behavior
per binding with the `idempotency` option.

## Correlation IDs

The runtime derives correlation metadata from the trigger source when possible:

- HTTP headers such as `x-correlation-id`
- event-native ids such as SQS `messageId` or EventBridge `id`
- source-specific fallbacks such as S3 `bucket:key:sequencer`

That correlation id is passed into handler metadata and stays consistent across guards, handler logic,
and after hooks.

## Batch failure behavior

- SQS returns partial batch failures through `batchItemFailures`
- Whole-batch retry sources rethrow by default
- `onRecordError()` can drop a failed record instead of retrying the whole batch when the trigger supports partial failure handling
