## Setup

Install and register the permissions plugin alongside auth:

```bash
bun add @lastshotlabs/slingshot-permissions
```

```typescript title="src/index.ts"
import { createServer } from '@lastshotlabs/slingshot';
import { createAuthPlugin } from '@lastshotlabs/slingshot-auth';
import { createPermissionsPlugin } from '@lastshotlabs/slingshot-permissions';

await createServer({
  port: 3000,
  security: { signing: { secret: process.env.JWT_SECRET! } },
  plugins: [
    createAuthPlugin({
      auth: { roles: ['user', 'editor', 'admin'], defaultRole: 'user' },
    }),
    createPermissionsPlugin(),
  ],
  packages: [blogPackage],
});
```

The permissions plugin provides a shared evaluator that all packages can use. It depends on
auth — auth identifies the actor, permissions decide what they can do.

## Grant patterns

Grants connect roles to actions. They are typically set during seed or admin operations.

### Simple action grants

```typescript
// All users can read posts
await permissions.grant({ role: 'user', action: 'post:read' });

// Editors can create and update posts
await permissions.grant({ role: 'editor', action: 'post:create' });
await permissions.grant({ role: 'editor', action: 'post:update' });

// Admins can do everything with posts
await permissions.grant({ role: 'admin', action: 'post:*' });
```

### Owner-scoped grants

Grants can be scoped so they only apply when the user owns the resource:

```typescript
// Users can update their own posts
await permissions.grant({
  role: 'user',
  action: 'post:update',
  scope: { authorId: '{{actor.id}}' },
});

// Users can delete their own posts
await permissions.grant({
  role: 'user',
  action: 'post:delete',
  scope: { authorId: '{{actor.id}}' },
});
```

`{{actor.id}}` is resolved at evaluation time to the current user's ID.

### Tenant-scoped grants

For multi-tenant apps, scope grants to the tenant:

```typescript
// Org members can read projects in their org
await permissions.grant({
  role: 'member',
  action: 'project:read',
  scope: { orgId: '{{actor.tenantId}}' },
});

// Org admins can manage projects
await permissions.grant({
  role: 'org-admin',
  action: 'project:*',
  scope: { orgId: '{{actor.tenantId}}' },
});
```

## Entity route policy

The most common way to use permissions is in entity route policy. The evaluator runs
automatically — no manual checks needed:

```typescript
const Post = defineEntity('Post', {
  namespace: 'blog',
  fields: {
    id: field.string({ primary: true, default: 'uuid' }),
    title: field.string(),
    body: field.string(),
    authorId: field.string(),
  },
  routes: {
    defaults: { auth: 'userAuth' },
    list: { auth: 'none' },
    get: { auth: 'none' },
    create: {
      permission: { requires: 'post:create' },
    },
    update: {
      permission: {
        requires: 'post:update',
        ownerField: 'authorId', // checks: actor owns this record
      },
    },
    delete: {
      permission: {
        requires: 'post:delete',
        ownerField: 'authorId',
        or: 'post:moderate', // OR: actor has moderator permission
      },
    },
  },
});
```

### How evaluation works

For `update` with `ownerField`:

1. Framework fetches the existing record
2. Checks: does the actor have `post:update`?
3. Checks: is `record.authorId === actor.id`?
4. Both must pass. If either fails, the request gets a `403`.

For `delete` with `or`:

1. First checks: `(has 'post:delete' AND is owner)`
2. If that fails, checks: `has 'post:moderate'`
3. If either path passes, the request proceeds.

## Package route permissions

Package routes use the same `permission` option:

```typescript
route.post({
  path: '/posts/:id/feature',
  auth: 'userAuth',
  permission: { requires: 'post:feature' },
  handler: async ({ params, respond }) => {
    await featurePost(params.id);
    return respond.json({ featured: true });
  },
});
```

### Scoped permissions on routes

Pass dynamic values from the URL into the permission check:

```typescript
route.post({
  path: '/orgs/:orgId/projects',
  auth: 'userAuth',
  permission: {
    requires: 'project:create',
    scope: { orgId: 'param:orgId' },
  },
  handler: async ({ body, params, respond }) => {
    const project = await createProject(params.orgId, body);
    return respond.json(project, 201);
  },
});
```

This checks: "Does this user have `project:create` scoped to this specific org?"

## Manual evaluation

When the declarative `permission` option is not flexible enough, use the evaluator directly:

```typescript
route.post({
  path: '/posts/:id/transfer',
  auth: 'userAuth',
  handler: async ({ actor, params, respond }) => {
    const post = await getPost(params.id);

    // Custom check: must own the post AND have transfer permission
    const canTransfer = await evaluator.can(
      { subjectId: actor.id!, subjectType: 'user' },
      'post:transfer',
      { authorId: post.authorId },
    );

    if (!canTransfer) return respond.json({ error: 'Forbidden' }, 403);

    await transferPost(post.id, params.newOwnerId);
    return respond.noContent();
  },
});
```

## WebSocket permissions

The same evaluator works for realtime subscriptions:

```typescript
ws: {
  endpoints: {
    '/ws': {
      onRoomSubscribe: async (ws, room) => {
        const allowed = await evaluator.can(
          { subjectId: ws.data.actorId, subjectType: 'user' },
          'room:join',
          { roomId: room },
        );
        return allowed;
      },
    },
  },
},
```
