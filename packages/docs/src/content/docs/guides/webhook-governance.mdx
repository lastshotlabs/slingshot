---
title: Webhook Governance
description: How registry-backed event definitions control webhook subscriptions, scope, and delivery
---

Webhook delivery in Slingshot is registry-governed. The webhook package no longer owns a separate
allowlist of string event names. Event owners define what can leave the process.

## The model

Every externally deliverable event starts with a definition:

```typescript
events.register(
  defineEvent('billing:invoice.paid', {
    ownerPlugin: 'billing',
    exposure: ['tenant-webhook'],
    resolveScope: (payload, ctx) => ({
      tenantId: payload.tenantId,
      actorId: ctx.actorId ?? null,
    }),
  }),
);
```

That definition decides:

- which exposure classes are allowed: `tenant-webhook`, `user-webhook`, or `app-webhook`
- how scope is projected
- whether external delivery is possible at all

If scope cannot be resolved for an externally exposed event, publish fails closed.

## Endpoint subscriptions

Webhook endpoint management uses `subscriptions`, not legacy `events` arrays:

```json
{
  "ownerType": "tenant",
  "ownerId": "tenant-a",
  "tenantId": "tenant-a",
  "url": "https://example.com/webhooks",
  "secret": "whsec_...",
  "subscriptions": [{ "pattern": "billing:invoice.*" }]
}
```

Patterns are normalized when the endpoint is created or updated. The stored record keeps concrete
approved event keys plus the optional `sourcePattern` they came from. That means:

- writes fail if a pattern does not resolve to any approved events
- future event registrations do not silently expand old endpoints
- reads and queue jobs operate on frozen concrete subscriptions

## Scope and authorization

Delivery is scope-aware by design:

- tenant-owned endpoints can only receive tenant-scoped webhook events for their tenant
- user-owned endpoints can only receive user-scoped webhook events for that user
- app-owned endpoints can only receive app-scoped webhook events for that app

Cross-tenant widening is rejected by the subscriber authorizer before delivery is queued.

## Delivery records

Webhook jobs and delivery records preserve the publish-time envelope identity:

- `event`
- `eventId`
- `occurredAt`
- subscriber identity
- source scope
- projected payload

Queues stay transport-focused. They carry the already projected payload and metadata; they do not
re-run authorization or rebuild scope from raw payload fragments.

## Migration note

Legacy stored endpoint rows are normalized at startup. If a row cannot be mapped safely to
`ownerType` / `ownerId` / concrete `subscriptions`, it is disabled instead of widened.
