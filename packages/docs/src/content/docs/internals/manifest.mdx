The manifest system turns a JSON file into a fully wired `createServer()` call. Know the internals when you're building custom plugins, registering a custom event bus, or debugging why a manifest option isn't applying.

## The pipeline

```
slingshot start
  │
  ├── 1. Read app.manifest.json
  ├── 2. Interpolate ${env:VAR} / ${VAR} → process.env values
  ├── 3. validateAppManifest() — Zod schema validation, type coercion
  ├── 4. Auto-load handlers (slingshot.handlers.ts or manifest `handlers` field)
  ├── 5. Auto-load first-party plugins via builtinPlugins dispatch table
  │      └── Resolve built-in strategy strings in plugin configs
  ├── 6. Auto-register built-in manifest event-bus adapters when selected
  │      └── Currently: BullMQ when manifest.eventBus selects "bullmq"
  ├── 7. manifestToAppConfig() — convert manifest to CreateServerConfig
  │      ├── Resolve built-in strategy strings → runtime functions
  │      ├── Resolve { handler: "name" } references → actual functions
  │      ├── Resolve ws.transport manifest specs → runtime transport adapters
  │      ├── Resolve ${importMetaDir} path placeholders
  │      └── Load TLS certs, initialize secrets provider, etc.
  └── 8. createServer(config) — standard boot from here
```

## Handler resolution

Any manifest field that maps to a function uses a handler reference:

```json
{
  "tenancy": {
    "resolution": "subdomain",
    "onResolve": { "handler": "resolveTenant" }
  },
  "logging": {
    "enabled": true,
    "onLog": { "handler": "onLog" }
  }
}
```

Handler names map to named exports from your handlers file (`slingshot.handlers.ts` by default). References resolve at startup, not at request time.

### The `handlers` manifest field

Control handler auto-loading with the top-level `handlers` field:

```json
{ "handlers": "src/handlers.ts" }
{ "handlers": { "dir": "src/handlers/" } }
{ "handlers": false }
```

- **String** — path to a single handlers file, relative to the manifest.
- **`{ "dir": "..." }`** — directory of handler files. All `.ts` and `.js` files (excluding `.d.ts`) are loaded in sorted order.
- **`false`** — disable handler auto-loading entirely.
- **Omitted** — defaults to `slingshot.handlers.ts` in the manifest's directory (loaded if it exists, silently skipped if not).

```typescript title="slingshot.handlers.ts"
declare const db: { tenants: { findById(id: string): Promise<unknown> } };
interface RequestLogEntry {
  readonly method?: string;
  readonly path?: string;
}
declare function sendToDatadog(entry: RequestLogEntry): Promise<void>;

export async function resolveTenant(id: string) {
  return db.tenants.findById(id);
}

export async function onLog(entry: RequestLogEntry) {
  await sendToDatadog(entry);
}
```

If a referenced handler isn't exported from the handlers file, `slingshot start` throws before serving any requests.

### The `hooks` export

In addition to handler functions, `slingshot.handlers.ts` can export a `hooks` object containing hook functions. These are separate from regular handler exports and are wired to lifecycle events declared in the manifest's `hooks` section.

```json title="app.manifest.json"
{
  "hooks": {
    "afterAdapters": [{ "handler": "captureAdapters" }]
  }
}
```

```typescript title="slingshot.handlers.ts"
import type { BareEntityAdapter } from '@lastshotlabs/slingshot-entity';

let transactionAdapter: BareEntityAdapter | null = null;

export const hooks = {
  captureAdapters(ctx: unknown) {
    const { adapters } = ctx as { adapters: Readonly<Record<string, BareEntityAdapter>> };
    transactionAdapter = adapters.Transaction ?? null;
  },
};
```

Each hook name in the manifest (e.g. `captureAdapters`) must match a key on the exported `hooks` object — not a top-level export. The `afterAdapters` hook runs after all entity adapters are created and passes `{ adapters }`, a frozen record of every entity adapter keyed by entity name. This is the primary way to capture adapters for cross-entity logic in custom operation handlers.

## Built-in strategies

Many manifest fields that accept a `{ "handler": "name" }` reference also accept a built-in strategy string. Strategy strings resolve to functions inside the framework — no handlers file needed.

### Framework-level strategies

| Field                             | Strategies                               | Description                                        |
| --------------------------------- | ---------------------------------------- | -------------------------------------------------- |
| `security.rateLimit.keyGenerator` | `"ip"`, `"user"`, `"ip+user"`            | Rate-limit key derivation                          |
| `security.rateLimit.skip`         | `"authenticated"`                        | Skip rate limits for logged-in users               |
| `logging.onLog`                   | `"json"`, `"pretty"`                     | Request log formatting                             |
| `metrics.normalizePath`           | `"strip-ids"`                            | Replace UUIDs and numeric path segments with `:id` |
| `validation.formatError`          | `"flat"`, `"grouped"`                    | Validation error response shape                    |
| `upload.authorization.authorize`  | `"owner"`, `"authenticated"`, `"public"` | Upload authorization policy                        |

### Plugin strategies

| Plugin                | Field                 | Strategies                                                      |
| --------------------- | --------------------- | --------------------------------------------------------------- |
| `slingshot-search`    | `tenantResolution`    | `"framework"` — reads `tenantId` from context                   |
| `slingshot-search`    | `adminGate`           | `"superAdmin"`, `"authenticated"`                               |
| `slingshot-admin`     | `accessProvider`      | `"slingshot-auth"` — auto-wires to auth plugin                  |
| `slingshot-admin`     | `managedUserProvider` | `"slingshot-auth"` — delegates user CRUD to auth adapter        |
| `slingshot-admin`     | `permissions`         | `"slingshot-permissions"` — reads from permissions plugin state |
| `slingshot-admin`     | `auditLog`            | `"memory"` — in-memory audit log for development                |
| `slingshot-webhooks`  | `queue`               | `"memory"` — in-memory webhook queue                            |
| `slingshot-community` | `authBridge`          | `"auto"` — bridges auth context to community principal          |
| SSR config            | `runtime`             | `"bun"`, `"node"`, `"edge"` — lazy-loads runtime package        |
| SSR config            | `isr.adapter`         | `"memory"` — in-memory ISR cache                                |

Strategy strings are resolved before the plugin factory runs. The plugin itself only sees the resolved function or object.

### Manifest field resolution order

When a manifest field can accept both a strategy string and a handler reference, the pipeline checks in this order:

1. If the value is a string matching a known strategy, resolve it to a function.
2. If the value is a `{ "handler": "name" }` object, look it up in the handler registry.
3. Otherwise, pass it through as-is.

## The handler registry

`createManifestHandlerRegistry()` is the programmatic interface to the manifest system. The CLI uses it internally — use it directly when you need to go beyond what the manifest file supports:

```typescript
import { createManifestHandlerRegistry, createServerFromManifest } from '@lastshotlabs/slingshot';
import type {
  SecretRepository,
  SlingshotEventBus,
  SlingshotPlugin,
} from '@lastshotlabs/slingshot-core';

declare const db: { tenants: { findById(id: string): Promise<unknown> } };
declare function createAnalyticsPlugin(config?: Record<string, unknown>): SlingshotPlugin;
declare function createRedisBus(config?: Record<string, unknown>): SlingshotEventBus;
declare function createVaultSecretProvider(config: Record<string, unknown>): SecretRepository;

const registry = createManifestHandlerRegistry();

// Register a handler function directly (instead of via handlers file)
registry.registerHandler('resolveTenant', () => async (id: string) => {
  return db.tenants.findById(id);
});

// Register a custom plugin that isn't in the built-in dispatch table
registry.registerPlugin('my-analytics-plugin', config => {
  return createAnalyticsPlugin(config);
});

// Register a custom event bus implementation
registry.registerEventBus('redis', () => {
  return createRedisBus({ url: process.env.REDIS_URL! });
});

// Register a custom secrets provider
registry.registerSecretProvider('vault', (config: Record<string, unknown>) => {
  return createVaultSecretProvider(config);
});

const server = await createServerFromManifest('./app.manifest.json', registry);
```

## The plugin dispatch table

Built-in first-party plugins are auto-loaded via a dispatch table (`builtinPlugins.ts`). The manifest references them by name:

```json
{ "plugin": "slingshot-auth", "config": { ... } }
```

`createServerFromManifest()` checks each plugin against the user-supplied registry first. If the plugin isn't registered, it looks up the package in the dispatch table and dynamically imports it. The table maps `"slingshot-auth"` → `createAuthPlugin` from `@lastshotlabs/slingshot-auth`, `"slingshot-permissions"` → `createPermissionsPlugin` from `@lastshotlabs/slingshot-permissions`, `"slingshot-entity"` → `createEntityPlugin` from `@lastshotlabs/slingshot-entity`, `"slingshot-game-engine"` → `createGameEnginePlugin` from `@lastshotlabs/slingshot-game-engine`, and so on.

If a referenced plugin is not in the dispatch table and not in the user registry, startup fails with a clear error. If it is in the dispatch table but its package isn't installed, the error tells you exactly which package to add.

Third-party plugins register via `registry.registerPlugin()` before you call `createServerFromManifest()`.

## Built-in event bus dispatch

`createServerFromManifest()` applies the same pattern to selected manifest event-bus literals. Today, `eventBus: "bullmq"` and the object form `{ "type": "bullmq", "config": { ... } }` auto-register the BullMQ adapter if the caller has not already registered a `bullmq` event bus in the registry.

That keeps BullMQ optional: the package is imported only when the manifest selects it. Custom buses such as Redis pub/sub, Kafka, RabbitMQ, or NATS still go through `registry.registerEventBus()`.

## Environment variable interpolation

Before validation, `createServerFromManifest()` walks the raw manifest and substitutes env var placeholders with `process.env` values. Two forms are supported:

```json
{
  "db": { "postgres": "${env:DATABASE_URL}" },
  "security": { "signing": { "secret": "${env:SIGNING_SECRET}" } },
  "hostname": "${APP_HOST}"
}
```

- `${env:NAME}` — explicit prefix form; preferred for clarity
- `${NAME}` — bare form; name must be all-uppercase letters, digits, and underscores

Lowercase tokens like `${importMetaDir}` are not touched — they go through a separate path-resolution step in `manifestToAppConfig()`.

If a referenced variable is not set in the environment, startup fails with:

```
Environment variable "DATABASE_URL" is not set (referenced in manifest at db.postgres)
```

## Path placeholders

The manifest uses `${importMetaDir}` to resolve paths relative to the manifest file:

```json
{
  "routesDir": "${importMetaDir}/src/routes",
  "workersDir": "${importMetaDir}/src/workers"
}
```

`manifestToAppConfig()` resolves them to absolute paths before passing to `createServer()`.

## Programmatic API

Three levels of control:

```text
// Level 1: fully automatic (CLI)
slingshot start

// Level 2: programmatic, auto-loads handlers file
import { createServerFromManifest } from '@lastshotlabs/slingshot';
const server = await createServerFromManifest('./app.manifest.json');

// Level 3: programmatic with custom registry
const registry = createManifestHandlerRegistry();
registry.registerPlugin('custom', createCustomPlugin);
const server = await createServerFromManifest('./app.manifest.json', registry);

// Level 4: manual pipeline (for testing, inspection)
import { validateAppManifest, manifestToAppConfig } from '@lastshotlabs/slingshot';
const raw = JSON.parse(readFileSync('./app.manifest.json', 'utf8'));
const manifest = validateAppManifest(raw); // throws on invalid config
const config = await manifestToAppConfig(raw, registry);
// config is now a standard CreateServerConfig — inspect it, modify it, pass it anywhere
```

## Dry run

Validate a manifest without starting the server:

```bash
slingshot start --dry-run
```

Runs `validateAppManifest()` and `manifestToAppConfig()`, logs the resolved config, and exits. Use it in CI to catch manifest errors before deploying.

## Validation notes

Manifest startup includes more than shape validation. Slingshot also performs cross-field checks for higher-level admin page config, including bad page references, invalid lookup operation names, and template placeholders that do not map to real entity fields.

Built-in plugin config is normalized before the factory runs. Strategy strings and handler refs are resolved into runtime objects first, so first-party plugins receive the same kinds of values in manifest mode that they would receive in code-first composition.
