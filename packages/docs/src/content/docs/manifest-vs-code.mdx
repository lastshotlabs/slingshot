---
title: Manifest vs Code
description: Code-first is the main docs path; manifests remain a supported alternate path
---

Both paths are supported. They do not deserve equal weight in the docs narrative.

## The recommendation

Start with code unless your app is mostly declarative assembly.

Use:

- code when you want to learn Slingshot's core public API surface
- manifests when you want a serialized configuration root with minimal code

## Why code leads the docs

The code-first path is where the framework's main authoring surfaces are most visible:

- `createServer()` and `createApp()`
- `definePackage(...)`
- `domain(...)` and `route.*(...)`
- `defineEntity(...)`
- `defineOperations(...)`
- `entity(...)`
- explicit event definitions and runtime escape hatches

That is the real framework model. The docs should teach it directly.

## Code path

```typescript
import { createServer, definePackage, domain, route } from '@lastshotlabs/slingshot';

const appPackage = definePackage({
  name: 'app',
  domains: [
    domain({
      name: 'health',
      routes: [
        route.get({
          path: '/health',
          auth: 'none',
          handler: ({ respond }) => respond.json({ ok: true }),
        }),
      ],
    }),
  ],
});

await createServer({
  packages: [appPackage],
});
```

## Manifest path

```json title="app.manifest.json"
{
  "manifestVersion": 1,
  "port": 3000,
  "plugins": [
    { "plugin": "slingshot-auth" }
  ]
}
```

```bash
slingshot start
```

## Choosing honestly

Use code when you are authoring the app's behavior.

Use a manifest when you are mostly selecting and configuring behavior that already exists.
