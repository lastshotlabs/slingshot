---
title: Manifest vs Code
description: Side-by-side comparison of the two app assembly paths
---

Both paths are supported and produce the same runtime behavior.

## Code-first

```typescript title="src/index.ts"
import { createServer, definePackage, domain, route } from '@lastshotlabs/slingshot';

const appPackage = definePackage({
  name: 'app',
  domains: [
    domain({
      name: 'health',
      routes: [
        route.get({
          path: '/health',
          auth: 'none',
          handler: ({ respond }) => respond.json({ ok: true }),
        }),
      ],
    }),
  ],
});

await createServer({
  packages: [appPackage],
});
```

Best when you are **authoring behavior**: custom routes, typed packages, entities, events,
and escape hatches. The full framework API surface is available.

## Manifest

```json title="app.manifest.json"
{
  "manifestVersion": 1,
  "port": 3000,
  "plugins": [{ "plugin": "slingshot-auth" }]
}
```

```bash
slingshot start
```

Best when you are **assembling and configuring**: selecting plugins, wiring databases,
setting security options. Custom behavior goes in `slingshot.handlers.ts`.

## When to switch

Start with whichever matches your app today. If a manifest app grows into substantial
custom authoring, migrate to a code-first app root. If a code-first app is mostly
configuration, a manifest can simplify it.
