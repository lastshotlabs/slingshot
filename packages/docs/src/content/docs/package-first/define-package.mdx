`definePackage(...)` is the canonical package boundary.

The examples on this page continue the [starter app](/slingshot/app-authoring/starter-app/) and its
`blogPackage`.

## A package you can actually ship

```typescript title="src/blog/package.ts"
import { z } from 'zod';
import { definePackage, domain, entity, route } from '@lastshotlabs/slingshot';
import { Post } from './post';
import { PostOperations } from './post-operations';

const previewBody = z.object({
  markdown: z.string(),
});

export const blogPackage = definePackage({
  name: 'blog',
  mountPath: '/blog',
  dependencies: ['slingshot-auth'],
  entities: [
    entity({
      config: Post,
      operations: PostOperations,
    }),
  ],
  middleware: {
    requireEditor: async (c, next) => {
      if (c.get('actor')?.role !== 'editor') {
        return c.json({ error: 'forbidden' }, 403);
      }
      await next();
    },
  },
  domains: [
    domain({
      name: 'preview',
      basePath: '/preview',
      routes: [
        route.post({
          path: '/',
          auth: 'userAuth',
          middleware: ['requireEditor'],
          request: { body: previewBody },
          responses: {
            200: { description: 'Rendered markdown preview' },
          },
          handler: ({ body, respond }) =>
            respond.json({
              html: `<article>${body.markdown}</article>`,
            }),
        }),
      ],
    }),
    domain({
      name: 'health',
      basePath: '/health',
      routes: [
        route.get({
          path: '/',
          auth: 'none',
          handler: ({ respond }) => respond.json({ ok: true, package: 'blog' }),
        }),
      ],
    }),
  ],
  publicPaths: ['/blog/health'],
});
```

That is a realistic package:

- entity-backed routes live in `entities`
- one-off but package-owned routes live in `domains`
- package-local middleware stays package-owned
- the app root only composes the package; it does not need to know the route internals

## What it declares

| Field               | Purpose                                                |
| ------------------- | ------------------------------------------------------ |
| `name`              | Stable package identity for ordering and diagnostics   |
| `mountPath`         | Base path shared by the package's entities and domains |
| `dependencies`      | Other packages or plugins that must be installed first |
| `entities`          | Package-owned entity modules                           |
| `domains`           | Package-owned non-entity route groups                  |
| `middleware`        | Named middleware available to package routes           |
| `capabilities`      | Published and required typed capabilities              |
| `tenantExemptPaths` | Paths that bypass tenant resolution                    |
| `csrfExemptPaths`   | Paths that bypass CSRF enforcement                     |
| `publicPaths`       | Paths the package exposes publicly                     |

## Why this exists instead of "just plugins"

Packages give the framework a stronger authoring contract:

- package-owned entities stay package-owned
- route and middleware naming stays package-owned
- capability wiring is explicit and typed
- apps compose product modules directly instead of hand-wiring lifecycle code

## Inspecting a package

Use `inspectPackage(...)` when you want the effective route and module graph without dropping into
framework internals.

```typescript
import { inspectPackage } from '@lastshotlabs/slingshot';
import { blogPackage } from './package';

const inspection = inspectPackage(blogPackage);

console.log(inspection.mountPath); // '/blog'
console.log(inspection.entities.map(entity => entity.resolvedPath));
console.log(inspection.domains.flatMap(domain => domain.routes));
```

That is useful for docs tooling, tests, and sanity checks when a package is large enough to need a
structural overview.
