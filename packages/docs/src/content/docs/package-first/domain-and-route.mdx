Not every package route should be generated from an entity.

For package-owned HTTP surfaces that are still part of the package contract, use `domain(...)` and
`route.*(...)`.

The examples on this page continue the [starter app](/slingshot/app-authoring/starter-app/) and
its `blogPackage`.

## A typed route that belongs to the package, not the entity

```typescript title="src/blog/package.ts"
import { z } from 'zod';
import { definePackage, domain, route } from '@lastshotlabs/slingshot';

const previewBody = z.object({
  markdown: z.string(),
});

const previewResponse = z.object({
  html: z.string(),
  wordCount: z.number(),
});

export const blogPackage = definePackage({
  name: 'blog',
  domains: [
    domain({
      name: 'preview',
      basePath: '/preview',
      services: {
        renderMarkdown(markdown: string) {
          return `<article>${markdown}</article>`;
        },
      },
      routes: [
        route.post({
          path: '/',
          auth: 'userAuth',
          summary: 'Preview rendered post content',
          request: { body: previewBody },
          responses: {
            201: {
              description: 'Rendered preview',
              schema: previewResponse,
            },
          },
          handler: async ({ body, services, respond }) => {
            const html = services.renderMarkdown(body.markdown);
            return respond.json(
              { html, wordCount: body.markdown.trim().split(/\s+/).filter(Boolean).length },
              201,
            );
          },
        }),
      ],
    }),
  ],
});
```

## What route definitions can declare

Package routes can carry:

- `auth`
- `permission`
- `rateLimit`
- `idempotency`
- `middleware`
- `event`
- typed `request`
- typed `responses`
- `summary` and `description`

That makes package routes a first-class authoring surface, not a low-level escape hatch.

## Domain-local services

`domain(...)` can also publish a service bag that route handlers receive as `ctx.services`.

```typescript
const postsDomain = domain({
  name: 'posts',
  services: {
    now: () => new Date().toISOString(),
  },
  routes: [
    route.withServices<{ now: () => string }>().get({
      path: '/clock',
      auth: 'none',
      handler: ({ services, respond }) => respond.json({ now: services.now() }),
    }),
  ],
});
```

Use this for domain-local helpers that belong to the route group itself.

## A route with params, permissions, and events

```typescript title="src/blog/package.ts"
const publishParams = z.object({
  id: z.string(),
});

route.post({
  path: '/posts/:id/publish-now',
  auth: 'userAuth',
  permission: { requires: 'blog:post.publish' },
  request: { params: publishParams },
  event: {
    key: 'blog:post.publishRequested',
    payload: ['id'],
  },
  handler: ({ params, respond }) =>
    respond.json({ queued: true, postId: params.id }),
});
```

That is the canonical package-route shape when the behavior belongs to the package but not to one
entity's generated route shell.

## When to use package routes vs entity routes

Use package routes when the behavior is:

- not naturally CRUD or named-op shaped
- not centered on one entity adapter
- still part of the package's public HTTP surface

Use the entity system when the route semantics should be generated from the domain model.
