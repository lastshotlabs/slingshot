Package-first authoring is the default. Escape hatches still matter.

## Raw plugins

Use `plugins` when you need direct lifecycle control:

- `setupMiddleware`
- `setupRoutes`
- `setupPost`
- `teardown`

That is still the right layer for some infrastructure integrations and lower-level framework work.

## Entity-level escape hatches

Inside the entity system, the main escape hatches are:

- [Generated-route executor overrides](/slingshot/entity-system/generated-routes-overrides-and-extra-routes/#override-a-generated-route) — replace the executor behind a CRUD or named-operation route
- [Extra routes](/slingshot/entity-system/generated-routes-overrides-and-extra-routes/#add-an-extra-route) — add routes alongside generated entity routes
- [Factory-backed adapter wiring](/slingshot/entity-system/storage-and-adapter-wiring/#factories) — custom repos while keeping the entity shell
- [Manual adapter wiring](/slingshot/entity-system/storage-and-adapter-wiring/#manual) — full control over adapter construction
- [Consumer field and naming conventions](/slingshot/entity-system/define-entity/#system-fields-storage-fields-and-conventions) — rename system fields, storage columns, and ID strategies
- [`createEntityPlugin()`](/slingshot/examples/config-driven-domain/#build-the-plugin) — lower-level orchestration work

These are part of the normal advanced authoring path, not internals. Full examples live in the linked pages.

## `inspectPackage(...)`

When a package gets large, `inspectPackage(...)` is a useful structural escape hatch. It lets you
reason about the effective package graph without reading framework internals directly.

## The rule

Stay on the package-first surface until you have a concrete reason to leave it.

Do not drop to raw plugins just because the plugin lifecycle existed first.
