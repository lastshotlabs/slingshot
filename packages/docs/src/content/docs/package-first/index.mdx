Package-first authoring is the main code-first surface for composing product modules in Slingshot.

This is the layer between the app root and the lower-level plugin lifecycle.

## Use this when

Use package-first authoring when you want to:

- scaffold an app from package modules
- keep non-entity routes package-owned
- publish typed cross-package capabilities
- compose entities and routes without dropping straight into raw plugin lifecycle code

## The core pieces

```typescript
import { definePackage, domain, route } from '@lastshotlabs/slingshot';

export const healthPackage = definePackage({
  name: 'health',
  domains: [
    domain({
      name: 'health',
      basePath: '/health',
      routes: [
        route.get({
          path: '/',
          auth: 'none',
          handler: ({ respond }) => respond.json({ ok: true }),
        }),
      ],
    }),
  ],
});
```

Package-first authoring is built around:

- `definePackage(...)`
- `domain(...)`
- `route.*(...)`
- `entity(...)`
- `defineCapability(...)`
- `provideCapability(...)`
- `entityRef(...)`

## What a package owns

A package can own:

- entity modules
- non-entity domain routes
- package-local middleware
- capability publication and requirements
- mount path and dependency metadata
- public, tenant-exempt, and CSRF-exempt paths

That is why packages are the right public abstraction for app builders.

## Progression: common to advanced

- common path: `definePackage(...)` plus `domain(...)` and `route.*(...)`
- common data path: attach `entity(...)` modules and move into the entity-system docs
- less common built-ins: typed capabilities, entity refs, package-local middleware, mount metadata
- escape hatches: raw plugins, lifecycle hooks, manual entity wiring, direct framework internals

## Where entities fit

The entity system plugs directly into package authoring through `entity(...)`. If a package owns a
data-shaped domain, the entity system is usually the canonical way to express it.

See [Entity System](/slingshot/entity-system/) next.
