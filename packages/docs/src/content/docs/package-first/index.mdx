Most backend apps start simple and get messy fast — routes scattered across files, middleware applied
inconsistently, shared logic duplicated between features. Packages solve this by giving each feature
area a clear boundary.

A package owns its routes, middleware, entities, and capabilities. The app root just composes them.

## What you'll learn

- How to define a package and mount it at a path
- How to write typed routes inside a package
- How to share capabilities between packages
- When to use packages vs raw plugins

## Your first package

```typescript
import { definePackage, domain, route } from '@lastshotlabs/slingshot';

export const healthPackage = definePackage({
  name: 'health',
  domains: [
    domain({
      name: 'health',
      basePath: '/health',
      routes: [
        route.get({
          path: '/',
          auth: 'none',
          handler: ({ respond }) => respond.json({ ok: true }),
        }),
      ],
    }),
  ],
});
```

Register it in your app:

```typescript
import { createServer } from '@lastshotlabs/slingshot';
import { healthPackage } from './packages/health';

await createServer({
  port: 3000,
  packages: [healthPackage],
});
```

That's it. The package mounts at `/health`, the route responds at `GET /health/`, and the app root
doesn't need to know anything about the route internals.

## When to use packages

| Situation                                       | Use                                                                         |
| ----------------------------------------------- | --------------------------------------------------------------------------- |
| Feature area with its own routes                | Package with `domain(...)`                                                  |
| Data-shaped domain (posts, users, orders)       | Package with `entity(...)` — see [Entity System](/slingshot/entity-system/) |
| Shared middleware or services                   | Package with capabilities                                                   |
| One-off infrastructure (custom health, metrics) | Package with a single domain                                                |
| Framework-level lifecycle work                  | Raw plugin — see [Escape Hatches](/slingshot/package-first/escape-hatches/) |

## Learn this section in order

1. [definePackage](/slingshot/package-first/define-package/) — declare the boundary, mount path, and dependencies
2. [domain and route](/slingshot/package-first/domain-and-route/) — write typed routes that belong to the package
3. [Capabilities and entityRef](/slingshot/package-first/capabilities-and-entity-ref/) — share services across packages

## Where entities fit

If your package owns data (posts, comments, orders), the entity system generates CRUD routes,
operations, and storage wiring from a single definition. You add `entity(...)` modules inside
`definePackage(...)`.

See [Entity System](/slingshot/entity-system/) next.
