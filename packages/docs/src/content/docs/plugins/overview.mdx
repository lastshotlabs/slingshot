This page is now the lower-level explanation of the plugin layer itself.

## The short version

- `packages` are the canonical code-first app-builder surface
- `plugins` are the lower-level lifecycle contract the framework still uses underneath

If you are building an app or authoring product modules, learn packages first.

If you are integrating with lifecycle hooks directly, read on.

## What the plugin layer is for

The plugin layer exists for:

- direct lifecycle hooks
- lower-level framework integrations
- compatibility with plugin-only packages
- escape-hatch work that does not fit the package authoring contracts

## The lifecycle

Plugins still run through these phases:

1. `setupMiddleware`
2. `setupRoutes`
3. `setupPost`
4. `teardown`

That contract still matters. It just should not be the first thing most users learn.

## How this relates to packages

`createApp()` compiles packages into framework plugins internally.

That is why the docs now separate:

- public package authoring
- lower-level plugin lifecycle reference
