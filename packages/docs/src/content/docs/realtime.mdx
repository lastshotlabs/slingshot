Slingshot supports two realtime transports: **WebSockets** for bidirectional communication and **Server-Sent Events (SSE)** for server-to-client streams. Both are configured in `createServer` and auto-discovered by the manifest.

## Pick the right transport

|               | WebSockets                                  | SSE                                   |
| ------------- | ------------------------------------------- | ------------------------------------- |
| Direction     | Bidirectional                               | Server → client only                  |
| Protocol      | `ws://` / `wss://`                          | `http://` / `https://`                |
| Browser API   | `new WebSocket(url)`                        | `new EventSource(url)`                |
| Auth          | Token in query param or first message       | Token in query param                  |
| Load balancer | Requires sticky sessions or Redis transport | Standard HTTP                         |
| Reconnection  | Manual                                      | Automatic (built into `EventSource`)  |
| Best for      | Chat, presence, collaborative editing       | Notifications, live feeds, dashboards |

**Use SSE** when data flows one way. The protocol is simpler, reconnection is automatic, and standard HTTP proxies handle it without special configuration.

**Use WebSockets** when clients send data back frequently — chat, multiplayer games, collaborative editors.

## WebSockets

### Configure WebSocket endpoints

<Tabs>
  <TabItem label="Manifest">
    ```json title="app.manifest.json"
    {
      "manifestVersion": 1,
      "port": 3000,
      "ws": {
        "endpoints": {
          "/ws": {
            "presence": true,
            "heartbeatIntervalMs": 30000
          }
        }
      }
    }
    ```
  </TabItem>
  <TabItem label="Code">
    ```typescript title="src/index.ts"
    import { createServer } from '@lastshotlabs/slingshot';

    await createServer({
      port: 3000,
      ws: {
        endpoints: {
          '/ws': {
            presence: true,
            heartbeatIntervalMs: 30000,
          },
        },
      },
    });
    ```

  </TabItem>
</Tabs>

Clients connect to `ws://localhost:3000/ws`. Room subscriptions, presence events, and message delivery work out of the box on a single instance.

### Rooms

Clients join rooms and publish messages to them. The server fans messages to all subscribers in that room:

```typescript
// Client-side
const ws = new WebSocket('ws://localhost:3000/ws?token=<jwt>');

ws.send(JSON.stringify({ type: 'subscribe', room: 'notifications:user-123' }));

ws.onmessage = event => {
  const data = JSON.parse(event.data);
  console.log(data);
};
```

```typescript
// Server-side — publish from a route or event handler
import { createApp, publish } from '@lastshotlabs/slingshot';
import { getContext } from '@lastshotlabs/slingshot-core';

const { app } = await createApp({});

app.post('/notifications', async c => {
  const ws = getContext(app).ws;
  if (ws) {
    publish(ws, '/ws', 'notifications:user-123', {
      type: 'new_notification',
      text: 'Hello',
    });
  }
  return c.json({ sent: true });
});
```

### Presence

When `presence: true`, the server tracks who's connected to each room and broadcasts join/leave events:

```json
{ "type": "presence_join", "room": "chat-room-1", "userId": "user-abc" }
{ "type": "presence_leave", "room": "chat-room-1", "userId": "user-abc" }
```

### Authentication

Native WebSocket upgrade requests don't support custom headers. Pass the JWT in the query string:

```typescript
const ws = new WebSocket(`ws://localhost:3000/ws?token=${jwt}`);
```

The upgrade handler validates the token and rejects the connection if it fails.

### Multi-instance scaling

By default, messages only reach clients connected to the same server instance. Add a Redis transport to fan out across instances:

<Tabs>
  <TabItem label="Manifest">
    ```json title="app.manifest.json"
    {
      "db": { "redis": "${env:REDIS_URL}" },
      "ws": {
        "transport": {
          "type": "redis",
          "options": {
            "connection": "${env:REDIS_URL}"
          }
        },
        "endpoints": { "/ws": { "presence": true } }
      }
    }
    ```
  </TabItem>
  <TabItem label="Code">
    ```typescript
    import { createRedisTransport, createServer } from '@lastshotlabs/slingshot';

    await createServer({
      ws: {
        transport: createRedisTransport({
          connection: process.env.REDIS_URL!,
        }),
        endpoints: {
          '/ws': { presence: true },
        },
      },
    });
    ```

  </TabItem>
</Tabs>

The transport uses Redis pub/sub to deliver messages to all instances. Each instance forwards them to its local subscribers. Room membership and presence remain in-memory per instance — the transport handles delivery only.

Run `slingshot infra check` to detect single-instance configuration gaps before you scale.

## Server-Sent Events

SSE streams JSON events from the server to connected clients over a plain HTTP connection. Clients use the native `EventSource` API — no library needed, no manual reconnect logic.

### Configure the SSE endpoint

<Tabs>
  <TabItem label="Manifest">
    ```json title="app.manifest.json"
    {
      "manifestVersion": 1,
      "port": 3000,
      "sse": {
        "endpoint": "/events",
        "heartbeatIntervalMs": 15000
      }
    }
    ```
  </TabItem>
  <TabItem label="Code">
    ```typescript title="src/index.ts"
    import { createServer } from '@lastshotlabs/slingshot';

    await createServer({
      port: 3000,
      sse: {
        endpoints: {
          '/events': {
            events: ['auth:login'],
            heartbeat: 15000,
          },
        },
      },
    });
    ```

  </TabItem>
</Tabs>

### Connect from the browser

`EventSource` doesn't support custom headers. Pass the JWT in the query string:

```typescript
// Client-side
const source = new EventSource(`/events?token=${jwt}`);

source.addEventListener('notification', event => {
  const data = JSON.parse(event.data);
  console.log('Notification:', data);
});

source.addEventListener('error', () => {
  // EventSource reconnects automatically after errors
  console.log('Reconnecting...');
});
```

### Publish events from the server

The SSE endpoint filters by the `clientSafeEvents` list — only events explicitly registered as safe reach clients. This prevents internal events (`security.*`, `auth:*`, etc.) from leaking.

```typescript
// Register which events are safe to forward (typically in a plugin's setupPost)
ctx.bus.registerClientSafeEvents(['post.created', 'post.updated', 'comment.created']);

// Emit an event from anywhere — clients subscribed to /events will receive it
ctx.bus.emit('post.created', { id: 'post-123', title: 'Hello' });
```

The SSE endpoint streams only registered client-safe events. Security-sensitive events never reach clients regardless of registration.

### Filter by event type

Clients can subscribe to specific event types via query parameter:

```typescript
// Only receive post.* events
const source = new EventSource(`/events?token=${jwt}&events=post.created,post.updated`);
```

<Aside type="caution">
  If you're using cookie-based sessions, cookies are sent with `EventSource` requests automatically
  — no token query param needed. Use this when building browser-only apps that use cookie auth.
</Aside>

## Plugin WebSocket endpoints

Slingshot plugins can self-wire WebSocket endpoints during `setupPost`. The `slingshot-community` and
`slingshot-game-engine` packages both use this pattern — they register incoming handlers, room
routing, and presence tracking on a named WS endpoint declared in your manifest or server config.
See the [WebSockets guide](/slingshot/guides/websockets/) for the entity channel config and self-wiring
patterns these plugins follow.
